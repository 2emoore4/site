midterm: concept, parts, and planning
evan moore - camwear
the concept

This project revolves around a system that will control the positioning of a camera, preferably a webcam or a smartphone that can act as a webcam. The main idea is that a user can control a camera remotely and view the feed in real time. Ideally, it will end up like a real life virtual reality system. With wearable computing devices on the horizon, it's possible for a system like this to have many applications.

the target audience

At first, I imagined this system being used in photography or film, in situations where it's not possible to have direct access to a camera (taking pictures with a Mars rover, filming from a camera mounted on a helicopter, etc). When I considered using it with wearable computers, I realized that there are significantly more applications. Someone could mount several of these cameras in their home and easily monitor them while on the road. Of course, what sets this apart from traditional webcams is the control system. The camera would react to head movements and the user would be able to see his home as if he were actually there. This system could also be used in teleconference type situations. You could stay at home while a moveable tablet takes your place in a meeting.

the technical system

I'll basically be dealing with one input and one output for this system. The input is human movement, and the output is camera movement. The camera can probably just be controlled by two servo motors. One motor will rotate the camera about an X axis, and the other motor will rotate it about a Y axis. I could even add a third motor to rotate it about a Z axis. This third one wouldn't be completely necessary, but it shouldn't be too hard to implement.

For the input, I need a sensor to detect human movement. I think it will probably be best to use an IMU, or inertial measurement unit. It's basically an accelerometer and a gyroscope built into one board. At first, the IMU will probably be connected to the motors via some very long wires, because I'm assuming it would be awfully difficult to come up with a way to implement wireless sensors.

Some of the work will be in code behind the scenes. I'll need to read the sensor data and convert it to motor movement.

Luckily, I don't have to worry about transmitting webcam data because I'll be using Android devices for both the camera, and for the remote display. There are already apps available for webcam servers and clients. I could possibly (as a stretch goal?) hook up the arduino to the android camera via usb in order to execute commands, like recording or taking pictures. Of course, that would require finding a way to read serial data through the android's usb port which is probably pretty difficult.

I might come up with more stretch goals in the future, but for now, here's a basic diagram of what I'll be doing.

the parts
  • arduino
  • android phone (for camera)
  • android tablet (for viewing camera feed)
  • tripod/camera mount (not sure how I'll do this yet)
  • wearable mount for IMU (probably eyeglass frames)
  • servo motor x3 @ $13.95 each
  • MPU-6050 IMU @ $39.95
the problems

The most difficult process that I forsee is reading the IMU sensor data. I'll need to find a way to combine the accelerometer with the gyroscope in order to create the most realistic movement. If this ends up being way over my head, I might be able to find a pre-existing library for reading IMU sensor data. Another tough part will be combining the two (or three) motors to get the desired movement, all while making sure it's not too bulky. If I end up trying to control the android phone with the arduino, I'll need to find a way to interface the two devices with serial communication, which might not even be possible.
